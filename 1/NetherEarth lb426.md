## 1\. Обзор проекта

Клон классической RTS *Nether Earth* с сохранением оригинальной механики, но с современной изометрической 3D-графикой, удобным управлением и модульной системой сборки роботов.  
**Сеттинг:** Астероид/планета Insignians. Игрок командует людскими силами против древней инопланетной расы.  
**Цель:** Захватить или уничтожить все вражеские Warbases.  
**Режим:** Одиночная игра, кампания/сценарии. Мультиплеер отложен на фазу 2.

## 2\. Игровые механики (Core Loop)

### 2.1 Летающий модуль игрока (Scout)

* Свободный полёт в 3D (WASD/стрелки + Q/E для высоты).
* Функции: разведка, выдача приказов, вход в структуры для строительства, ручной контроль юнита (удержание `Ctrl` + ЛКМ).
* Коллизии: прозрачен для своих юнитов, отталкивается от ландшафта и врагов.

### 2.2 Модульная сборка роботов

Каждый робот состоит из компонентов, привязанных к сущности ECS:

|Тип модуля|Ограничения|Влияние|
|-|-|-|
|Шасси (Chassis)|Ровно 1|Скорость, проходимость, тип поверхности|
|Оружие (Weapon)|1–3 слота|Урон, дальность, скорострельность|
|Электроника|0–1|Радар, ИИ-ускорение, точность|
|Ядерный заряд|0–1|Радиус 8 миль, самоуничтожение после применения|

**Правила валидации:**  
✅ Обязательно: 1 шасси + ≥1 оружие  
✅ Остальное опционально, но ограничено слотами и ресурсами  
❌ Неверные комбинации блокируются в UI с пояснением

### 2.3 Приказы и ИИ

* **Seek \& Destroy:** Автономный поиск и уничтожение ближайших вражеских юнитов/структур.
* **Seek \& Capture:** Захват нейтральных/вражеских фабрик или баз.
* **Patrol / Defend / Idle:** Статическое поведение.
* **Manual Control:** Временный override ИИ. По завершении приказа или потере связи юнит возвращается к последнему автономному приказу или переходит в `Idle`.
* **Мульти-выделение:** Прямоугольное выделение (`Shift` + ЛКМ/тянуть) + группы `Ctrl+1..0`.

### 2.4 Структуры

|Тип|Функция|Захват|Уничтожение|
|-|-|-|-|
|Factory|Производит 1 специфический ресурс + General|Длительное нахождение юнита в зоне входа (прогресс-бар)|Только ядерным зарядом|
|Warbase|Сборка роботов из захваченных ресурсов|Аналогично фабрике|Только ядерным зарядом|
|Препятствия|Скалы, каньоны, кратеры|Недоступны|Разрушаемы (опц.)|

### 2.5 Ресурсы и экономика

* 7 типов ресурсов: `Electronics`, `Nuclear`, `Phasers`, `Missile`, `Cannon`, `Chassis`, `General`.
* **Тик производства:** 1 игровые сутки = 30 сек реального времени (настраивается в конфиге).
* Фабрики бездействуют, пока не захвачены.
* `General` генерируется всеми захваченными фабриками пропорционально `production\_rate \* general\_ratio`.
* Ресурсы списываются мгновенно при старте постройки робота.

### 2.6 Карта

* Размер: `512 × 512` игровых единиц (тайлов).
* Формат уровня: `map\_data.ron` (координаты структур, препятствия, спавны).
* Границы карты жёсткие, выход за пределы блокируется.

### 2.7 Условия победы/поражения

* **Победа:** Захвачены или уничтожены все вражеские Warbases.
* **Поражение:** Уничтожены все собственные Warbases.

## 3.4. Ключевые подсистемы

1. **Система фабрик (`Factory`)**  
Отвечает за накопление ресурсов и производство новых роботов. Каждая фабрика имеет очередь сборки, скорость производства, владельца. Фабрики могут захватываться. При захвате генерируют ресурсы (5 единиц/день).
2. **Система сборки роботов (`Robot Builder`)**  
UI/логика, позволяющая выбирать модули и создавать `Robot`. Хранит конфигурацию бота (список компонентов). По сабмиту создает сущность-робота с нужными компонентами (`Chassis`, `Weapons`, `Electronics` и т.д.) и спавнит её у фабрики.
3. **AI-система (`AI`)**  
Управляет роботами противника. Простые конечные автоматы или поиск пути (DFS/BFS) позволяют им искать цели: ближайшие вражеские базы, нейтральные фабрики или атаковать игрока. По умолчанию ИИ выдаёт заказы аналогично игроку (атаковать, захватывать).
4. **Карта и коллизии (`Map/Physics`)**  
Содержит данные уровня: местоположение препятствий, баз, границы карты. Система физики обрабатывает движение роботов: ходьба по поверхности, столкновения с объектами, ограничение движения (обход скал/каньонов). Может использовать простой grid/A\* для навигации роботов в режиме «найди путь».
5. **UI/Frontend**  
Графические экраны (главное меню, HUD, экран сборки роботов). Реализуется через встроенные средства Bevy (`bevy\\\_ui`) или отдельный GUI (`egui`). Показывает информацию об игроке: ресурсы, количество роботов, мини-карта/радар. Подсистема GUI взаимодействует с геймплей-логикой (например, при клике «Построить робота» вызывает создание сущности).
6. **Audio**  
Проигрывает звуковые эффекты (стрельба, взрывы) и музыку. Интегрируется через аудио-API (`rodio`/`bevy\\\_audio`).
7. **Сетевая подсистема (опционально)**  
Если планируется мультиплеер, включает синхронизацию состояния, отправку приоритетных событий (передвижение курсора, приказы, респауны роботов). Может быть реализована поверх Tokio (TCP/UDP).
8. **Прочее**  
Логика сохранений/загрузок (сериализация состояния), конфигурации (настройки, управление).

## 3.6. Форматы данных

* **Конфигурации:** Предпочтительны читабельные форматы (`RON`, `TOML`, `YAML`) для настроек игры (управление, графика, звук). `RON` хорошо интегрирован с Serde и похож на Rust-синтаксис, `TOML` знаком по Cargo.
* **Сохранения:** Сериализуют состояние игры (список фабрик, роботов, ресурсов). Можно использовать `Serde + RON` (или `bincode` для компактности, но `RON` обеспечивает читаемость). Важно сохранить версии формата для обратной совместимости.
* **Данные уровня:** Карта, стартовое расположение фабрик и баз. Можно хранить как `JSON`/`TOML`/`RON`-файл или бинарные ресурсы. Рекомендуется `RON` для простоты редактирования (описание координат, типов фабрик).

## 3.7. Локализация

* **Языки:** Русский (RU) и Английский (EN). Интерфейс и тексты игры должны поддерживать оба языка (следует предусмотреть механизм переключения и загрузки строк из файлов).
* **Ресурсы:** Для переводов (строки, подписи UI) использовать форматы `i18n` (например, файлы JSON или RON-таблицы). Невыделенные по умолчанию элементы интерфейса обозначить как «не указано», чтобы локализацию можно было добавить позднее.

## 3.8. Тестирование

* **Unit-тесты:** Проверка базовых функций (сборка робота из модулей, захват фабрики, логика команд, сериализация данных). Rust имеет встроенный фреймворк тестов `#\\\[test]`.
* **Integration-тесты:** Тестирование взаимодействия подсистем, например эмуляция простого сценария (игра начинается, строится робот, он захватывает фабрику). Можно использовать headless-режим без рендеринга для таких тестов.
* **Автоматизация:** Автотесты запускаются в CI при каждом коммите, покрывают ключевые сценарии.
* **Регрессии:** Добавление тестов при обнаружении багов, чтобы избежать повторения.

## 3.9. Проектирование компонентов Bevy ECS

Для реализации модульной системы в рамках архитектуры Bevy ECS необходимо правильно спроектировать компоненты и системы.

### 1\. Компоненты, описывающие модули

Для каждого типа модулей следует создать структуры, содержащие их характеристики. Эти структуры затем будут использоваться как компоненты для сущности робота.

```rust
struct LocomotionModule { speed: f32, mobility: f32 }
// Хранит характеристики шасси, такие как максимальная скорость и проходимость.

struct WeaponModule { damage: f32, range: f32, fire\\\_rate: f32, weapon\\\_type: WeaponType }
// Хранит характеристики единичного оружейного модуля. weapon\\\_type — enum (Cannon, Missiles, Phasers).

struct ElectronicsModule { radar\\\_range: f32, ai\\\_pathfinding: bool }
// Хранит характеристики модулей электроники.

struct NuclearModule { explosion\\\_radius: f32, is\\\_equipped: bool }
// Хранит информацию о наличии и параметрах ядерного заряда.


2\. Компоненты, объединяющие робота

&#x09;Сама сущность робота будет группировать эти модули.

```rust

struct Robot { 

&#x20;   modules: Vec<Box<dyn ModuleTrait>>, 

&#x20;   total\_cost: u32, 

&#x20;   current\_health: f32, 

&#x20;   max\_health: f32 

}



Примечание: Использование динамических трейтов (dyn ModuleTrait) может обеспечить гибкость, позволяя роботу иметь разнородные модули в едином списке, однако в чистом Bevy ECS рекомендуется выносить каждый модуль в отдельный компонент для лучшей кэш-локальности и параллелизма.





3.10. Логика работы системы (Системы Bevy)

Система валидации сборки: Перед созданием нового робота запускается система, проверяющая наличие ровно одного LocomotionModule и как минимум одного WeaponModule. При провале робот не создаётся, игроку выводится ошибка.

Система расчёта характеристик: После валидации система просматривает модули робота и рассчитывает итоговые параметры (скорость из LocomotionModule, урон как сумма damage \* fire\_rate всех WeaponModule и т.д.). Обновляет производные компоненты: Speed, TotalDamage, Weight.

Система управления производством: При заказе робота проверяется наличие ресурсов. Если хватает — списывает их, запускает таймер постройки. При нехватке — отклоняет заказ.

3.11. Масштабируемость и конфигурация

## 4\. Архитектура кода

Проект строится вокруг **Entity Component System (ECS)** Bevy — это наиболее подходящий подход для RTS с большим количеством автономных юнитов.

### 4.1. Основные принципы архитектуры

* **Data-oriented design**: вся игровая логика описывается через компоненты и системы.
* **Плагины Bevy**: каждый крупный модуль игры — отдельный плагин (`Plugin`).
* **События (Events)**: для коммуникации между системами (приказы, захват, уничтожение и т.д.).
* **Ресурсы (Resources)**: глобальные данные (текущие ресурсы игрока, состояние игры, настройки).
* **Состояния (States)**: `AppState` / `GameState` (`Menu`, `Playing`, `Paused`, `GameOver` и т.д.).
* **Минимальная мутация**: системы должны быть как можно более параллельными. Избегать `\\\&mut World`, использовать `Query` и `Commands`.

### 4.2. Основные плагины (рекомендуемая структура)

1. `CorePlugin` — базовые вещи (камера, input, время игры).
2. `PlayerModulePlugin` — управление летающим модулем игрока.
3. `RobotPlugin` — всё, что касается роботов (сборка, компоненты, движение, бой).
4. `StructurePlugin` — фабрики, warbases, захват.
5. `ResourcePlugin` — экономика, производство ресурсов.
6. `CommandPlugin` — система приказов роботам (`Seek \\\& Destroy`, `Seek \\\& Capture` и т.д.).
7. `AIPlugin` — автономное поведение роботов.
8. `CombatPlugin` — боевая система.
9. `UiPlugin` — пользовательский интерфейс (мини-карта, панель ресурсов, меню строительства).
10. `SaveLoadPlugin` — сохранение и загрузка.
11. `AudioPlugin` — звуки и музыка.

### 4.3. Ключевые компоненты (Components)

**Общие для всех сущностей:**

* `Transform`, `GlobalTransform`, `Velocity`, `Health`, `Team` (`Player` / `Enemy` / `Neutral`), `Selectable`, `Hoverable`

**Специфические:**

* `PlayerModule` — маркер для летающего модуля игрока + компоненты управления.
* `Robot`:

  * `Chassis` (`enum`: `Bipod`, `Tracks`, `AntiGrav`) + характеристики (скорость, проходимость, может ли летать).
  * `Weapons` (`Vec` или отдельные компоненты: `Cannon`, `Missiles`, `Phasers`) с параметрами урона, дальности, перезарядки.
  * `Electronics` (`Radar`, `Shield`, `Accuracy` и т.д.).
  * `NuclearBomb` (опционально).
* `Structure` (`enum`: `Factory`, `Warbase`):

  * `StructureType`, `Owner`, `ProductionRate`, `CaptureProgress`.
* `Command` (текущий приказ робота):

  * `enum RobotCommand`: `Idle`, `MoveTo(Vec3)`, `SeekAndDestroy`, `SeekAndCapture(Entity)`, `Defend`, `ManualControl`.
* `Pathfinding` / `Navigation` (A\* или steering behaviors).
* `ProductionQueue` (для warbase).

### 4.4. Ключевые системы (Systems) и порядок выполнения

**Startup / Setup системы:**

* `setup\\\_camera`, `spawn\\\_player\\\_module`, `spawn\\\_initial\\\_structures`.

**Update системы (в порядке приоритета):**

1. **Input systems** — обработка клавиатуры/мыши для модуля и приказов.
2. **Player Module movement** — полёт и взаимодействие с объектами (вход в фабрику/базу).
3. **Command Issuing** — выдача приказов роботам через модуль.
4. **AI / Robot Behavior** — выполнение текущего приказа (движение, поиск цели, захват, атака).
5. **Movement \& Navigation** — pathfinding и физическое перемещение.
6. **Combat** — обнаружение целей, стрельба, нанесение урона.
7. **Capture System** — прогресс захвата структур.
8. **Resource Production** — начисление ресурсов с фабрик/баз.
9. **Robot Construction** — UI и спавн новых роботов в warbase.
10. **Health \& Death** — проверка здоровья, уничтожение сущностей.
11. **UI Update** — обновление интерфейса (ресурсы, мини-карта).

> 💡 `FixedUpdate` рекомендуется использовать для физики, AI и боевой логики, чтобы поведение было детерминированным и стабильным при разном FPS.

### 4.5. Дополнительные рекомендации

* Использовать `bevy\\\_mod\\\_picking` (или аналог) для выбора роботов и структур.
* Для камеры — орбитальная/изометрическая камера с зумом и следованием за модулем.
* События: `RobotCommandIssued`, `StructureCaptured`, `RobotDestroyed`, `ResourcesChanged`.
* Для сохранения — сериализация через `bevy\\\_scene` или `ron` / `serde`.
* Избегать `\\\&mut World` где возможно — предпочитать `Queries` и `Commands`.
* Логика роботов должна быть полностью автономной после выдачи приказа (как в оригинале).

## 5\. Требования к ассетам и форматам

### 5.1. Графика

* **Основной стиль:** Low-poly 3D с изометрической/forced-perspective камерой (рекомендуется) или пиксель-арт в высоком разрешении.
* **Форматы:**

  * Модели: `.gltf` / `.glb` (с анимациями).
  * Текстуры: `.png` (с поддержкой PBR где возможно).
  * Анимации: `idle`, `move`, `attack`, `death` для каждого типа шасси и оружия.
* **Размер роботов:** значительно крупнее летающего модуля (как «дома» в оригинале).
* **Минимальный набор ассетов:**

  * Летающий модуль игрока (3–4 варианта окраски).
  * 3 шасси + комбинации с оружием (минимум 9–12 визуальных вариантов роботов).
  * Фабрики (3–4 типа), Warbases (4 основные + нейтральные).
  * Эффекты: выстрелы (пули, ракеты, лазеры), взрывы, захват (анимация флага/маркера).

### 5.2. Аудио

* **Формат:** `.ogg`
* **Набор:** движение (разные для каждого шасси), выстрелы (3 типа), взрывы, захват, UI-клики, фоновая музыка в ретро-sci-fi стиле.

### 5.3. Другие требования

* Все ассеты должны быть лицензированы для открытого использования (или созданы самостоятельно).
* Поддержка замены ассетов для моддинга (через папку `assets/`).
* **Оптимизация:** LOD для дальних роботов, GPU Instancing, Object Pooling для эффектов/снарядов.

## 7\. План тестирования

### 7.1. Виды тестирования

* **Unit-тесты:** компоненты ECS (движение, расчёт урона, стоимость строительства).
* **Integration-тесты:** взаимодействие систем (выдача приказа → выполнение AI).
* **Manual gameplay testing:**

  * Полёт модуля и выдача приказов.
  * Сборка роботов всех возможных комбинаций.
  * Захват фабрик и баз (таймер захвата).
  * Бои между разными конфигурациями роботов.
  * Победа/поражение на оригинальных сценариях.
* **Performance testing:** 60 FPS при 50+ роботах на карте.
* **Balance testing:** сравнение силы разных комбинаций модулей.

### 7.2. Тестовые сценарии

1. Базовый: постройка простого робота → приказ `Seek \\\& Destroy`.
2. Полный цикл: захват фабрики → накопление ресурсов → постройка армии → атака базы.
3. Крайние случаи: робот только с `Cannon`, только с `Anti-Grav` + 3 `Phasers`, использование ядерной бомбы.
4. Сохранение/загрузка на разных этапах.

### 7.3. Инструменты

* `cargo test` + встроенные инструменты Bevy.
* Встроенный debug-режим (отображение pathfinding, здоровья, приказов).
* Логирование ключевых событий (`tracing` / `log`).
